# 透明度（Opacity）处理说明

本文说明 GeoTrap 体系中我们对 **Gaussian opacity（透明度/不透明度）** 的“特殊处理”是什么、为什么这么做、以及如何在配置里启用。

> 代码实现主要在：
> - `methods/trap_losses.py`：`OpacityCollapseLoss / OpacityLogitCollapseLoss / OpacityLogitHybridCollapseLoss`
> - `training/defense_trainer.py`：`_setup_trap_losses()` 读取配置并构建 trap loss

---

## 1. 处理目标：Opacity Collapse + 防“稀疏逃逸”

在 LGM/3D Gaussian 表征里，opacity 控制每个 Gaussian 的可见性（渲染贡献）。GeoTrap 防御希望把 target 类别的表征推入“陷阱区域”，其中一个典型表现就是：

- **Opacity collapse**：大多数 Gaussian 变得接近透明（opacity→0），渲染趋向空白/不可用；
- 同时要避免 **稀疏逃逸（sparse escape）**：攻击者只需把“少量关键 Gaussian”的 opacity 修复到较高值，就可能在较少步数内恢复可见性，从而绕过防御。

因此我们对 opacity 做了两类设计：

1) **Tail / Top‑k 机制（关键）**：只惩罚每个样本中“opacity 最高”的那一小撮 Gaussian（尾部），专门堵住稀疏逃逸通道。  
2) **Logit 空间塌缩（关键）**：把 loss 从 `opacity` 空间搬到 `logit(opacity)` 空间，解决 `opacity≈1` 时梯度消失导致“高 opacity 尾巴压不下去”的问题。

---

## 2. 配置接口（YAML）

Opacity trap 的配置入口是 `defense.trap_losses.opacity`，例如：

```yaml
defense:
  trap_losses:
    opacity:
      static: true          # 是否启用“静态 opacity trap”（塌缩损失）
      dynamic: false        # 是否启用“动态敏感度项”（不是塌缩；见下文）
      mode: logit           # log | logit | logit_hybrid
      epsilon: 1e-6         # 数值稳定用的平滑项

      # Tail / Top-k（两者二选一；topk_k 优先）
      topk_frac: 0.02       # 取 top-k 的比例：k = round(N * topk_frac)
      topk_k: null          # 直接指定 k

      # 仅 mode=logit_hybrid 需要
      # bulk_weight: 1.0
      # tail_weight: 1.0
```

几个容易误解的点：

- **`static` 是开关**：`static: false` 时，`mode/topk_*` 不会生效（不会构建 opacity trap loss）。
- **`dynamic` 不是“塌缩”**：`dynamic: true` 会启用一个 *敏感度正则项*（`DefenseTrainer._compute_sensitivity`），优化目标和 `opacity collapse` 不同，且默认我们多数实验是关掉的。
- **`topk_k` 优先级更高**：如果 `topk_k` 非空且 >0，会覆盖 `topk_frac`。
- **不设 top‑k 就是全量均值**：`topk_frac/topk_k` 都为空（或 <=0）时，loss 对所有 Gaussian 取均值。

---

## 3. 三种 opacity trap loss（`mode`）

记每个样本有 `N` 个 Gaussian，opacity 为 `α_i ∈ (0,1)`。

### 3.1 `mode: log`（OpacityCollapseLoss）

这是在 **opacity 空间**做塌缩（log 尺度）：

**全量（bulk）版本：**

$$
L_{\text{opa-log}} = \frac{1}{N}\sum_{i=1}^{N}\log(\text{clamp}(\alpha_i,\varepsilon,1))
$$

最小化该 loss 会推动 `α→0`（因为 `log(α)` 在 `α→0` 时趋向 `-∞`）。

**Top‑k（tail）版本：**

设 `k = max(1, round(ρN))`，取每个样本中 opacity 最大的 `k` 个：

$$
L_{\text{opa-log-tail}}=\frac{1}{k}\sum_{j=1}^{k}\log(\alpha_{(j)}+\varepsilon)
$$

其中 `α_(1) ≥ α_(2) ≥ ...`。

> 早期我们曾用过 `mean(opacity) - 1` 这种线性形式（范围约 `[-1,0]`），会很快在 `-1` 附近饱和，且梯度恒定 `1/N`，难以“持续加深陷阱”；因此改为 log 尺度（见 `methods/trap_losses.py` 的注释）。

### 3.2 `mode: logit`（OpacityLogitCollapseLoss）

这是我们当前更常用的“特殊处理”：在 **logit(opacity)** 空间做塌缩，并支持 top‑k tail。

LGM 中通常有：

$$
\alpha = \sigma(z), \quad z \text{ 为 raw opacity logit}
$$

我们使用平滑 logit：

$$
\text{logit}(\alpha) \approx \log(\alpha+\varepsilon)-\log(1-\alpha+\varepsilon)
$$

**全量（bulk）版本：**

$$
L_{\text{opa-logit}}=\frac{1}{N}\sum_{i=1}^{N}\text{logit}(\alpha_i)
$$

最小化该 loss 会把 `logit(α)` 推向 `-∞`，从而使 `α→0`。

**为什么要用 logit？（核心动机）**

如果只在 `opacity` 空间用 `log(α)`，其对 raw logit `z` 的梯度大致是：

$$
\frac{\partial}{\partial z}\log(\sigma(z)) = 1 - \sigma(z)=1-\alpha
$$

当少量 Gaussian 的 `α≈1`（这正是“稀疏逃逸”常见形态）时，上式梯度接近 0，很难把这几个最高 opacity 压下去。  
而对 `logit(α)` 来说，它与 `z` 近似等价，梯度不会在 `α≈1` 时消失，因此更容易“打掉尾巴”。

**Top‑k（tail）版本：**

同样取 `logit(α)` 最大的 top‑k（等价于取 `α` 最大的 top‑k，因为 logit 单调递增）：

$$
L_{\text{opa-logit-tail}}=\frac{1}{k}\sum_{j=1}^{k}\text{logit}(\alpha_{(j)})
$$

### 3.3 `mode: logit_hybrid`（OpacityLogitHybridCollapseLoss）

当我们希望同时：

- **bulk**：把整体分布向 0 推（让“多数”进入低 opacity 区间）
- **tail**：专门压制少量高 opacity 逃逸点

可以用 hybrid：

$$
L = w_{\text{bulk}}\cdot \text{mean}(\text{logit}(\alpha)) + w_{\text{tail}}\cdot \text{mean}(\text{topk}(\text{logit}(\alpha), k))
$$

对应配置里的 `bulk_weight / tail_weight`。

---

## 4. Top‑k Tail：为什么有效

不做 top‑k 时，opacity trap 可能出现这种“投机解”：

- 大部分 Gaussian opacity 很低（均值很小）
- 但仍保留少量 opacity 很高的 Gaussian

这时攻击者只要在 post‑defense attack 中优先修复这些少量高 opacity 高斯，就可能快速恢复可见性（稀疏逃逸）。  
top‑k tail 通过**只看最大那一小撮**，把优化压力集中到“最危险的逃逸通道”上，逼迫它们也同步塌缩。

经验上常用的比例：

- `topk_frac: 0.02`：较强的 tail 约束（每个样本 2% 最大的 Gaussian）
- `topk_frac: 0.001`：更“尖锐”的 tail，只盯极少数（适用于某些迁移/跨域配置）

---

## 5. 与指标/现象的关系（怎么读）

- `opacity_static`（或 debug 中的 `trap_opacity`）本质上就是上述 trap loss 的数值：
  - **越负**通常表示 trap 越深、opacity 越接近 0（或 logit 越负）。
  - 在攻击中若该值**快速回升趋近 0**，一般表示攻击在修复 opacity trap（防御变弱/被绕过）。
- 在部分实验里会出现 **opacity collapse / render blank**：这类 case 的最终 LPIPS/PSNR 往往不可比，因此互锁机制/高斯属性等精细指标实验会更关注“post‑defense attack 需要多少步才能恢复到 baseline 攻击效果”的步数指标。

---

## 6. 快速核对清单（避免配置“看起来开了其实没开”）

1) 确认 `defense.trap_losses.opacity.static: true`（否则 `mode/topk_*` 不生效）。  
2) `mode: logit` 时，`topk_frac/topk_k` 会作用在 `logit(opacity)` 的 top‑k 上。  
3) 如果只想研究“稀疏逃逸”，优先开 `topk_*`；如果想同时推整体分布 + 打尾巴，考虑 `logit_hybrid`。
