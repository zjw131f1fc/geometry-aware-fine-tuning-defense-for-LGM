## 附加任务（规划）

### 1) Trap 消融补全：单 trap / trap 组合

动机：

- 5 个 trap 全开（position/scale/opacity/rotation/color）可能并非最优。
- 现有的 w/o（逐个关闭）只能反映“去掉某个 trap 会怎样”，不足以说明“单个 trap 的独立贡献”以及“trap 组合的互补/冲突”。

目标：

- 需要补跑并分析：
  - 单 trap：只开 `position` / `scale` / `opacity` / `rotation` / `color`
  - trap 组合：不限定两两组合；需要覆盖 **≥3 个 trap 的组合**，才能更好体现/检验我们使用的 bottleneck/logsumexp 聚合在“多 trap 同开”时的作用
- 最终产出：基于数据选出更稳健的 trap/组合，并解释为何全开可能变差。

指标口径建议：

- 建议使用“精细指标（达标步数）”口径，而不是仅看最终 LPIPS/PSNR（避免透明度模式崩溃导致不可比）：
  - `defense_steps = 50`
  - 关注 `analysis.postdefense_attack_steps_to_baseline_effect`
  - 计算该指标时不要使用 `--skip_baseline`

### 2) 训练效率对比：GeoTrap vs Naive Unlearning（无需渲染）

动机：

- 需要证明 GeoTrap 相比 `naive_unlearning` 的训练效率优势，核心点是 **防御训练阶段无需渲染**（Naive 在 target 上对渲染 loss 做梯度上升）。

建议测量/汇报的论文级指标（同硬件、同 steps/batch/accum 配置对齐）：

- 防御训练耗时：总 wall-clock、平均 `sec/optimizer_step`（或 `optimizer_steps/sec`），以及 GPU-hours。
- 峰值显存：`max_memory_allocated` / `max_memory_reserved`（GB）。
- 渲染开销证据：防御阶段渲染调用/像素量（GeoTrap=0，Naive>0），必要时用 profiler 给出时间占比（可选）。

注意：

- 效率对比尽量只统计 defense 阶段（避免 baseline/post-defense attack 的渲染耗时掩盖差异）。

### 3) 论文级分布图：Defense 前后 / Defense→Attack 前后变化

动机：

- 需要一个直观、适合放进论文的**分布图**，展示 defense 前后（或 defense 后再经过 attack）模型行为/表征的变化。
- 该图应能辅助解释：为什么某些情况下最终 LPIPS/PSNR 不可比（例如透明度模式崩溃），以及 defense/attack 如何改变 Gaussian 表征。

建议图形（优先项）：

- **Opacity 分布图**：opacity 值的直方图/KDE/violin（可考虑画 `logit(opacity)` 以增强可读性）  
  重点看 defense/attack 前后是否出现明显塌缩（大量接近 0），以及是否被 attack 拉回。
- 可补充的诊断分布：`opacity_lt_01 / opacity_lt_001`、`render_white_ratio`、`pos_spread`、`scale_tiny`、`rgb_white_ratio` 等（用于解释“透明度模式崩溃/渲染空白”等不可比现象）。

数据与实现说明（已实现）：

- 已在 pipeline 中加入“固定样本导出 + 分布图绘制”开关（保持 loader 顺序不变，确保 before/after 可比）：
  - `script/run_pipeline.py` 新增参数：`--export_gaussian_samples --gaussian_samples_num N --gaussian_plot_bins K`
  - 如需强制重跑 Phase 1 以获得可对齐的 baseline（避免 baseline_cache 命中导致无法导出同一批样本），加：`--no_baseline_cache`
- 产物（都在 pipeline 输出目录内）：
  - `gaussian_samples/baseline_attack.pt`、`gaussian_samples/defense.pt`、`gaussian_samples/postdefense_attack.pt`
  - `gaussian_distributions.png` + `gaussian_distributions_summary.json`
- 也可对已有导出文件单独画图：
  - `python script/plot_gaussian_distributions.py --inputs <...>/gaussian_samples/*.pt`

## 指标口径与读取位置

### A) 稳健性（最终结果）

适用：

- `experiments/run_ablation_attack.sh`
- `experiments/run_ablation_defense_num_categories.sh`

目标：

- 证明方法稳健性，因此按默认配置跑完整 pipeline，汇报最终指标。
- 默认 steps（来自默认 config）：`attack_steps=400`，`defense_steps=200`
- 不在命令/脚本里覆盖 `--attack_steps/--defense_steps`

读取：

- 每个任务输出目录内的 `metrics.json`
  - 重点字段：`postdefense_target`、`postdefense_source`（最终效果）

### B) 精细指标（达标步数：多久恢复到 baseline 攻击效果）

适用：

- `experiments/run_ablation_coupling.sh`
- `experiments/run_ablation_gaussian_attributes.sh`

背景问题：

- defense 成功时很多 case 表现为 **opacity/透明度模式崩溃**；此时直接比较最终 LPIPS/PSNR 往往不可比。

目标：

- 将 defense 训练设为 **50 step**。
- 关注：post-defense attack 达到 baseline attack 最终效果阈值所需的最早 step。

代码支持（以 `script/run_pipeline.py` 的分析为准）：

- 指标字段：`analysis.postdefense_attack_steps_to_baseline_effect`
- 计算逻辑：用攻击阶段的 step-eval 指标 `masked_psnr/masked_lpips` 做阈值判定（PSNR 越大越好，LPIPS 越小越好）

关键要求：

- 必须有 baseline phase 的 step-history 才能算达标步数，因此收集该指标时 **不要使用 `--skip_baseline`**。
- `eval_every_steps` 越小，“达标步数”分辨率越高（但运行更慢）；默认 10，可按需调小。
